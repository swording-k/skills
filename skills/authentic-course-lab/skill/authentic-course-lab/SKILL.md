# 真实课程实验

把实验指导书视为唯一验收标准。目标不是“写出一份像完成了的报告”，而是按照指导书真实完成实验，并为每一项要求留下可核验的证据。

## 最高优先级原则

1. **先读要求，再做实验。** 必须完整阅读指导书、任务书、评分标准、报告模板和补充材料。
2. **逐条完成，不擅自改题。** 不得为了容易完成而缩小、替换或弱化要求。
3. **真实操作，不伪造过程。** 必须真实运行代码、操作指定软件、使用实际数据或按要求操作设备。
4. **每一步都要可证明。** 按指导书步骤执行，并在关键状态完成后立即记录和截图。
5. **截图必须来自真实界面。** 禁止生成、绘制、拼接、重建或渲染截图冒充操作结果。
6. **报告只能写已经做过的内容。** 不得先编写成功结果，再寻找或制造对应证据。
7. **失败结果也要诚实记录。** 若实验要求“尝试”，真实尝试后未成功也是有效结果，应分析原因，不得伪造成功。
8. **未逐项验收，不得宣布完成。**

开始实验前阅读：

- [证据与截图标准](references/evidence-standard.md)
- [逐步实验记录模板](references/step-record-template.md)

## 通用工作流

### 第一阶段：解析实验指导书

完整提取以下信息：

- 实验目的；
- 实验环境、软件、设备和版本要求；
- 每一个编号步骤；
- 每一步的输入、操作和预期现象；
- 必须自行实现的内容和禁止调用的工具；
- 测试次数、测试数据和对比条件；
- 要求提交的源代码、工程文件、数据、截图和报告；
- 报告格式与必写章节；
- 教师现场检查时可能要求演示的功能。

建立“要求—执行—证据”矩阵：

| 编号 | 指导书原始要求 | 计划如何执行 | 需要保存的证据 | 当前状态 |
|---|---|---|---|---|

必须保留指导书原意。遇到模糊要求时，从课程目标和上下文判断；若不同理解会显著影响结果，再询问用户。

### 第二阶段：检查实验环境

在操作前确认：

- 指定软件、IDE、仿真器、数据库、虚拟机或设备是否存在；
- 软件是否在系统目录、用户目录或其他常见位置；
- 所需权限、插件、驱动、依赖和数据集是否可用；
- 当前工程和用户已有文件，避免覆盖用户工作；
- 截图和实验输出保存位置。

不能因为命令行找不到软件就直接声称未安装。应搜索应用、打开软件并检查实际界面。

遇到管理员密码、设备接线、实验室硬件操作、危险化学操作等必须由用户完成的步骤时：

1. 先完成所有可准备工作；
2. 明确告诉用户下一步具体做什么；
3. 等用户完成后继续；
4. 不保存密码或敏感信息。

### 第三阶段：制定逐步执行与截图计划

把指导书每一步拆成可执行单元。每个单元至少写清：

| 字段 | 内容 |
|---|---|
| 步骤编号 | 与指导书对应 |
| 操作前状态 | 当前界面、文件或设备状态 |
| 具体操作 | 点击、输入、运行、连接或测量动作 |
| 输入数据 | 必须使用的值或文件 |
| 预期结果 | 应出现的现象或输出 |
| 实际结果 | 完成后记录 |
| 截图时机 | 哪个状态出现后截图 |
| 证据文件 | 截图、数据、工程或日志路径 |

截图计划必须覆盖：

- 实验环境或软件启动状态；
- 每个主要操作步骤；
- 关键参数设置；
- 关键中间结果；
- 最终结果；
- 对比实验、异常输入或参数变化结果；
- 指导书明确要求截图的每一处。

不要无意义地每次点击都截图；但任何影响实验结论的步骤都必须有截图或同等强度的证据。

### 第四阶段：真实完成实验

严格按步骤执行：

1. 执行一个步骤；
2. 检查实际结果是否符合预期；
3. 若失败，使用系统化调试找出原因；
4. 成功后立即保存工程、数据和原生截图；
5. 在实验记录中写下实际输入、输出、现象和问题；
6. 再进入下一步。

根据实验类型选择真实操作方式：

- **编程实验：** 实现真实逻辑，接受新输入，运行测试，不能固定打印结果。
- **数据库实验：** 在指定数据库工具中真实建库、建表、执行 SQL，并保存查询结果。
- **电子电路/EDA 实验：** 在指定软件中真实搭建电路、设置元件、运行仿真并保存波形或工程。
- **MATLAB/Simulink/控制实验：** 真实建立模型、设置参数、运行仿真并保存曲线和模型文件。
- **操作系统/虚拟机实验：** 在指定系统或虚拟机中执行要求的命令与配置。
- **网络实验：** 真实产生流量、抓取和分析；只有指导书要求时才使用抓包文件。
- **数据分析/机器学习实验：** 使用真实数据，执行预处理、训练、评估和可视化，保留指标和配置。
- **物理、化学、机械等实验：** 使用指导书指定的仿真软件、数据或设备；需要人工设备操作时与用户协作。

不得用终端、脚本或其他工具替代指导书明确指定的 GUI 软件，除非指导书允许，或只是用于辅助验证。

### 第五阶段：验证真实性与完整性

完成实验后，逐项检查：

- 更换输入或参数后，结果是否真实变化；
- 是否实现了指导书要求的所有功能；
- 是否满足指定测试次数、数据范围和对比条件；
- 指定软件是否真的操作过；
- 每一步是否有对应记录和证据；
- 截图是否来自真实操作状态；
- 工程文件和数据能否重新打开；
- 结论是否由实际结果支持。

对于编程实验，优先加入自动测试；但测试通过不能替代 GUI、仿真器或指定软件的实际操作证据。

### 第六阶段：根据真实证据编写实验报告

报告必须让没有参与实验的人也能按照内容复现实验。使用 [实验报告写作指南](references/report-guide.md)。

报告至少包含：

1. 封面和基本信息；
2. 实验名称与目的；
3. 实验环境、软件、设备和数据；
4. 实验原理；
5. 实验设计；
6. **详细操作步骤：每一步写明“做什么、怎么做、输入什么、得到什么”；**
7. **每个主要步骤紧跟对应真实截图和图注；**
8. 关键代码、配置、公式或电路参数；
9. 实际结果、表格、曲线和对比分析；
10. 遇到的问题、排查过程与解决方法；
11. 实验结论。

截图不能集中堆放而不解释。每张截图前后必须说明：

- 这一步为什么要做；
- 具体如何操作；
- 截图中应观察什么；
- 它证明了哪一项实验要求。

### 第七阶段：准备教师现场演示

使用 [教师现场演示指南](references/live-demo-guide.md) 为用户编写一份可以照着执行的说明，包括：

- 从哪个文件或软件开始；
- 每一步在哪里点击、输入什么；
- 使用哪组稳定的演示数据；
- 每一步预期显示什么；
- 如何证明不是固定输出或伪造结果；
- 如何演示参数改变、异常输入或对比结果；
- 老师可能提问什么，应如何解释；
- 若现场环境变化，如何快速恢复。

### 第八阶段：最终验收与交付

运行：

```bash
python3 scripts/lab_evidence_audit.py <实验工作目录>
```

再使用 [最终完成验收清单](references/completion-audit.md) 手动逐项核对。

最终交付目录通常包含：

- 实验报告；
- 源代码或工程文件；
- 原始实验数据和结果文件；
- 原生截图；
- 必要的启动脚本；
- 教师现场演示说明；
- 指导书要求的其他文件。

删除或隔离过期版本、临时文件、模拟证据和可能误交的错误报告。

## 工具协作

- 需要 Word、PDF、表格、幻灯片时，使用对应文档技能。
- 需要操作本机 GUI 时，使用 Computer Use 或指定软件的专用工具。
- 需要操作网页时，使用浏览器工具；网页工具不能替代指定桌面软件。
- 遇到失败或异常行为时，使用系统化调试。
- 实现算法或程序功能时，使用测试驱动开发。

## 必须立即纠正的错误模式

- 未完整阅读指导书就开始实现；
- 把固定示例输出当成可交互程序；
- 用生成图片冒充实验截图；
- 用终端输出代替指导书指定软件的操作截图；
- 为了得到漂亮结果而伪造数据、曲线、测量值或成功状态；
- 偷偷降低实验难度或改变测试条件；
- 只展示最终结果，不记录中间步骤；
- 报告中写了实际未执行的操作；
- 只说“测试通过”，却没有逐条证明指导书要求；
- 没有告诉用户现场检查时如何操作和演示。
