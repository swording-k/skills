# 证据与截图标准

## 证据强度

优先使用最能证明真实完成状态的证据：

1. 指定软件中的实际状态，加上可重新打开的工程、数据或结果文件；
2. 完成操作后的原生应用窗口截图；
3. 指导书明确要求的命令输出、日志、波形、曲线、查询结果或测量数据；
4. 自动测试、校验程序或计算复核；
5. 文字说明只能辅助解释，不能单独证明完成。

## 合格截图

合格截图必须：

- 来自真实操作完成后的应用、终端、仿真器、虚拟机或设备界面；
- 显示本步骤涉及的关键输入、设置、中间状态或结果；
- 能够与指导书中的某一个步骤对应；
- 使用系统截图、应用截图或计算机操作工具直接捕获；
- 没有经过生成、重绘、模拟或内容拼接；
- 与报告中的描述和数值一致；
- 尽量避免无关窗口、通知和隐私内容。

优先截取应用窗口。若必须截全屏，应在使用前检查画面。

## 不合格证据

- 根据文本重新渲染出的“截图”；
- 仿照软件界面制作的图片；
- 实际没有执行却人为构造的输出、数据、曲线、波形、日志或测量值；
- 操作前截图冒充操作后结果；
- 用其他工具替代指导书指定软件的截图；
- 从文档、网络、其他学生或其他机器复制的结果；
- 截图数值与报告描述不一致。

## 每一步的证据记录

为每个主要步骤记录：

- 步骤编号；
- 操作时间或顺序；
- 使用的软件、工程、数据或设备；
- 输入值和参数；
- 实际操作；
- 实际结果；
- 截图或结果文件路径；
- 此证据证明的指导书要求。

## 文件命名建议

使用能说明步骤和来源的名称：

- `step-01-software-started-actual.png`
- `step-02-parameters-set-actual.png`
- `step-03-simulation-result-actual.png`
- `experiment-data-original.csv`
- `final-project.slx`

若实验明确要求模拟或生成数据，应清楚标注其用途。否则，发现文件名包含 `fake`、`mock`、`generated`、`simulated`、`伪造` 等词时，应检查其真实性并避免误交。
